# GRU From-Scratch Benchmark

This package implements recurrent cells from scratch and benchmarks three
architectures on synthetic long-memory tasks:

- **RNN**: vanilla Elman recurrent cell, used as the ungated baseline.
- **GRU**: both a readable naive cell and a fused cell that matches PyTorch's
  stacked-parameter layout.
- **LSTM**: fused LSTM cell with tuple state `(h, c)`.

The benchmarks compare how well each model carries information across many
timesteps. The two datasets, **Adding** and **Copy Memory**, are generated by
the code itself on the selected device. There is no external data file to
download, add, or copy into the repository.

## Datasets

Both benchmark datasets are synthetic. They are created every training step by
`make_adding_batch(...)` or `make_copy_batch(...)`, directly on CPU, CUDA, or
MPS depending on the device chosen by the driver.

This means:

- there are no `.csv`, `.npz`, or downloaded dataset files;
- every run can sample fresh examples indefinitely;
- the same `torch.Generator` seed is used so RNN, GRU, and LSTM see the same
  data stream during a comparison;
- CUDA runs avoid repeated host-to-device dataset copies because tensors are
  created on the CUDA device.

You can inspect the generated data with:

```bash
.venv/bin/python -m gru.tasks.adding
.venv/bin/python -m gru.tasks.copy_memory
```

### Adding Problem

The Adding task is a regression benchmark for selective long-range memory.

For batch size `B` and sequence length `T`, `make_adding_batch` returns:

- `x`: shape `(B, T, 2)`
- `y`: shape `(B, 1)`

The input has two channels:

- `x[..., 0]`: random values sampled from `Uniform(0, 1)`
- `x[..., 1]`: a binary mask with exactly two `1` values per sequence

The target is the sum of the two values selected by the mask:

```text
y = x[t1, 0] + x[t2, 0]
```

A model that always predicts the mean target value `1.0` gets MSE `1/6`, or
about `0.1667`. A useful model should move clearly below that baseline.

Recommended progression:

```bash
.venv/bin/python -m gru.train_adding --seq_len 50 --steps 1000
.venv/bin/python -m gru.train_adding --seq_len 100 --steps 2000
.venv/bin/python -m gru.train_adding --seq_len 200 --steps 4000
```

As `seq_len` grows, the RNN has to backpropagate through more repeated `tanh`
transitions. The gated models usually remain trainable for longer sequences.

### Copy Memory

The Copy Memory task is a sequence-to-sequence classification benchmark. The
model must memorize `K` symbols, hold them through a blank delay of length `T`,
then output the original symbols after a go cue.

`make_copy_batch` returns:

- `x`: shape `(B, T + 2K, 10)`, one-hot input tokens
- `y`: shape `(B, T + 2K)`, integer class targets

Vocabulary:

- `0`: blank token
- `1..8`: data symbols
- `9`: go cue

Input layout:

```text
[K data symbols] [T-1 blanks] [go cue] [K blanks]
```

Target layout:

```text
[K+T blanks] [same K data symbols]
```

The naive baseline predicts blanks everywhere and guesses uniformly among the
eight data symbols in the final answer slots. Its cross-entropy is:

```text
(K / (T + 2K)) * log(8)
```

Recommended progression:

```bash
.venv/bin/python -m gru.train_copy --K 5 --T 20 --steps 2000
.venv/bin/python -m gru.train_copy --K 10 --T 50 --steps 4000
.venv/bin/python -m gru.train_copy --K 10 --T 100 --steps 8000
```

The important metric is answer-position accuracy on the last `K` tokens. Loss
below the baseline means the model has started to remember the message; high
answer accuracy means it is copying the sequence in the correct order.

## Benchmark Tutorial

The training drivers support the same core options:

```bash
.venv/bin/python -m gru.train_adding \
  --seq_len 100 \
  --hidden_size 64 \
  --steps 2000 \
  --batch_size 64 \
  --lr 1e-3 \
  --eval_every 100 \
  --models rnn gru lstm
```

```bash
.venv/bin/python -m gru.train_copy \
  --K 10 \
  --T 50 \
  --hidden_size 64 \
  --steps 4000 \
  --batch_size 64 \
  --lr 1e-3 \
  --eval_every 100 \
  --models rnn gru lstm
```

Use `--models` to run a subset:

```bash
.venv/bin/python -m gru.train_adding --models gru lstm
.venv/bin/python -m gru.train_copy --models rnn gru
```

Difficulty knobs:

| Task | Option | Meaning |
| --- | --- | --- |
| Adding | `--seq_len` | Number of timesteps the model must process |
| Copy Memory | `--K` | Number of symbols to remember |
| Copy Memory | `--T` | Length of the blank delay |
| Both | `--hidden_size` | Recurrent state size |
| Both | `--steps` | Optimization budget |
| Both | `--batch_size` | Number of samples per update |
| Both | `--lr` | Adam learning rate |

What to compare:

| Metric | Adding | Copy Memory |
| --- | --- | --- |
| Primary loss | MSE | cross-entropy per position |
| Baseline | constant `1.0`, MSE `0.1667` | constant blank plus uniform answer guess |
| Extra metric | final MSE | final answer accuracy |
| Efficiency | wall-clock seconds | wall-clock seconds |
| Size | parameter count | parameter count |

For fair scientific comparisons, remember that equal `hidden_size` does not
mean equal parameter count. GRU uses about 3x the recurrent parameters of an
RNN at the same hidden size, and LSTM uses about 4x. The drivers report the
parameter count so you can account for this when reading the summary table.

## How Models Are Built

`RecurrentSeq` is the shared sequence wrapper. It accepts a cell factory and
runs that cell over an input tensor shaped `(B, T, D)`.

Every cell follows this interface:

```python
def forward(self, x_t, state):
    ...

def init_state(self, batch_size, device, dtype):
    ...

@staticmethod
def state_to_h(state):
    ...
```

RNN and GRU use a single tensor state `h`. LSTM uses tuple state `(h, c)`.
`RecurrentSeq` treats the state as opaque and asks `state_to_h(...)` for the
hidden tensor used by the output head.

Readout modes:

| Readout | Output shape | Used for |
| --- | --- | --- |
| `last` | `(B, out_size)` | Adding problem |
| `mean` | `(B, out_size)` | available for pooled sequence tasks |
| `all` | `(B, T, out_size)` | Copy Memory |

The Adding driver uses `readout="last"` because the task has one scalar target.
The Copy Memory driver uses `readout="all"` because it predicts a token at
every timestep.

## GRU Naive vs GRU Fused

Both GRU implementations compute the same PyTorch-style equations:

```text
r_t = sigmoid(W_ir x_t + b_ir + W_hr h_{t-1} + b_hr)
z_t = sigmoid(W_iz x_t + b_iz + W_hz h_{t-1} + b_hz)
n_t = tanh(W_in x_t + b_in + r_t * (W_hn h_{t-1} + b_hn))
h_t = (1 - z_t) * n_t + z_t * h_{t-1}
```

The reset gate placement matters. PyTorch applies `r_t` after the hidden
linear transform:

```text
r_t * (W_hn h_{t-1} + b_hn)
```

Applying the reset gate directly to `h` first, as in `W_hn (r_t * h)`, is a
different model and will not match `torch.nn.GRUCell`.

### Naive GRU

`GRUCellNaive` stores six separate matrices:

```text
W_ir, W_iz, W_in
W_hr, W_hz, W_hn
```

It is useful for learning, debugging, and ablations because each gate is
visible as separate code.

### Fused GRU

`GRUCellFused` stores stacked matrices:

```text
weight_ih: (3H, input_size)
weight_hh: (3H, hidden_size)
bias_ih:   (3H)
bias_hh:   (3H)
```

Each timestep performs two larger linear operations:

```python
gi = F.linear(x, weight_ih, bias_ih)  # (B, 3H)
gh = F.linear(h, weight_hh, bias_hh)  # (B, 3H)
i_r, i_z, i_n = gi.chunk(3, dim=-1)
h_r, h_z, h_n = gh.chunk(3, dim=-1)
```

This is the same layout used by PyTorch's `GRUCell`: gate order is
`[reset, update, new]`.

### Why Fusion Is Better On CUDA

Fusion does not change the GRU math. It changes how the math is scheduled on
hardware.

Naive GRU does six smaller matrix multiplies per timestep: three for the input
side and three for the hidden side. Fused GRU does two larger matrix multiplies
per timestep: one stacked input projection and one stacked hidden projection.

This matters especially on CUDA:

1. **Fewer kernel launches**

   A CUDA operation has launch overhead. In a recurrent model the loop over time
   is inherently sequential, so this overhead is paid at every timestep. Six
   small matmuls plus elementwise ops create much more launch overhead than two
   larger matmuls plus the same gate activations.

2. **Better GPU occupancy**

   GPUs are efficient when work is large enough to keep many threads busy.
   Several small matrix multiplies often underuse the device. Stacking the gate
   weights creates larger matrix operations that are easier for cuBLAS/cuDNN to
   schedule efficiently.

3. **Better memory locality**

   The fused layout reads contiguous blocks of gate weights and produces one
   contiguous `(B, 3H)` activation tensor. The naive layout repeatedly launches
   independent operations over smaller tensors, which increases memory traffic
   and intermediate tensor handling.

4. **Same parameters, cleaner copying**

   Because the fused layout matches PyTorch's layout, `copy_from_torch_cell`
   can directly copy `weight_ih`, `weight_hh`, `bias_ih`, and `bias_hh`. This
   makes numerical equivalence testing straightforward.

On CPU, fusion can still help, but the speedup may be smaller. On CUDA, the
benefit is usually more visible because launch overhead and device occupancy are
central performance bottlenecks for timestep-by-timestep Python recurrent code.

## Equivalence Tests

The test files instantiate PyTorch reference cells, copy their parameters into
the scratch implementation, then compare forward outputs and gradients in
`float64`.

```bash
.venv/bin/python -m gru.test_gru_cell
.venv/bin/python -m gru.test_lstm_cell
```

Passing these tests means the scratch cells are not just qualitatively similar
to PyTorch. They use the same parameter layout, gate order, bias convention, and
reset-gate placement.

The project currently verifies:

- `GRUCellNaive` vs `torch.nn.GRUCell`
- `GRUCellFused` vs `torch.nn.GRUCell`
- `GRUCellNaive` vs `GRUCellFused`
- `LSTMCellFused` vs `torch.nn.LSTMCell`
- forward outputs and input/state gradients

## Equation Reference

### Vanilla RNN

```text
h_t = tanh(W_ih x_t + b_ih + W_hh h_{t-1} + b_hh)
```

Parameter count:

```text
H * input_size + H * H + 2H
```

### GRU

```text
r_t = sigmoid(W_ir x_t + b_ir + W_hr h_{t-1} + b_hr)
z_t = sigmoid(W_iz x_t + b_iz + W_hz h_{t-1} + b_hz)
n_t = tanh(W_in x_t + b_in + r_t * (W_hn h_{t-1} + b_hn))
h_t = (1 - z_t) * n_t + z_t * h_{t-1}
```

Parameter count:

```text
3H * input_size + 3H * H + 6H
```

### LSTM

```text
i_t = sigmoid(W_ii x_t + b_ii + W_hi h_{t-1} + b_hi)
f_t = sigmoid(W_if x_t + b_if + W_hf h_{t-1} + b_hf)
g_t = tanh(W_ig x_t + b_ig + W_hg h_{t-1} + b_hg)
o_t = sigmoid(W_io x_t + b_io + W_ho h_{t-1} + b_ho)
c_t = f_t * c_{t-1} + i_t * g_t
h_t = o_t * tanh(c_t)
```

Parameter count:

```text
4H * input_size + 4H * H + 8H
```

## Extending The Benchmark

To add a new recurrent cell:

1. Implement the cell in `gru/cells/`.
2. Provide `forward`, `init_state`, and `state_to_h`.
3. Add the class to `gru/cells/__init__.py`.
4. Add a branch in `build_model(...)` inside the training driver.
5. Add an equivalence test if there is a PyTorch reference implementation.

To add a new task:

1. Add a batch generator under `gru/tasks/`.
2. Return tensors already on the requested device.
3. Pick the correct readout mode: `last` for one target per sequence, `all`
   for one target per timestep.
4. Create a training driver or adapt one of the existing drivers.
5. Include a simple baseline so benchmark plots are interpretable.
